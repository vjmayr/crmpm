# Decision Log

- 2026-07-06: `User` model's table is named `users`, not `user`. Reason: `user` is a reserved word in PostgreSQL. Future FKs to User (e.g. `manager_id`, `created_by`, `assignee_id`) must reference `users.id`.
- 2026-07-06: Login form validates email with a regex (`app/core/forms.py`), not WTForms' `Email()` validator. Reason: `Email()` requires the optional `email_validator` package, which is not in the approved dependency list.
- 2026-07-06: No PostgreSQL driver (`psycopg2`/`psycopg`) is installed yet. Reason: not in the approved dependency list. `ProductionConfig` reads `DATABASE_URL` but cannot connect to Postgres until a driver is added and approved — needed before any production deploy.
- 2026-07-06: `templates/` lives at the repo root, not Flask's default `app/templates/`. Reason: matches the Phase 0 kickoff spec's `templates/base.html` path; required an explicit `template_folder` override in `create_app()` (`app/__init__.py`).
- 2026-07-06: Enabled global `CSRFProtect` (`app/extensions.py`, `app/__init__.py`) instead of relying on per-`FlaskForm` CSRF only. Reason: Phase 1's inline qualification-status edit is a plain HTMX POST with no `FlaskForm`, so it had zero CSRF protection under the Phase 0 setup. `base.html`'s `<body hx-headers='{"X-CSRFToken": ...}'>` now covers every HTMX request app-wide; `WTF_CSRF_ENABLED=False` in `TestingConfig` still disables it for tests. Extends the approved dependency set only in the sense of using a Flask-WTF feature (`CSRFProtect`) not previously wired up — no new package.
- 2026-07-06: Organization delete is blocked with a route-level count check (`Person.query.filter_by(organization_id=...).count()`), not a DB-level `ON DELETE RESTRICT`. Reason: explicitly sanctioned by the Phase 1 kickoff spec ("blocked ... or soft-prevented in the route"); simpler and identical behavior across SQLite dev and Postgres prod.
- 2026-07-06: `permission_to_contact` is NOT a guard in `promote_to_lead()` (`app/crm/services.py`). Reason: the flag governs contact affordances, not pipeline membership — a qualified prospect whose permission is still pending may be tracked as a Lead. Intentional design, per the Phase 2a kickoff spec.
- 2026-07-06: `PromotionError`/`LeadStateError` render as HTTP 200 with an inline error partial (HTMX requests) or a flash + redirect (plain requests) — never a 4xx/5xx. Reason: consistent with the existing WTForms validation-failure convention (also 200); keeps HTMX's default swap behavior working, since htmx doesn't swap non-2xx responses without extra config. Establishes the pattern future service-exception surfacing (Offer FSM, Phase 4) should follow.
- 2026-07-06: The promote-to-lead action uses the `HX-Redirect` response header (full client-side navigation to the new Lead's detail page) rather than an in-place swap. Reason: promotion creates a new resource the user should land on, unlike the qualification-edit and close-lead actions, which update state on the same page and use `hx-swap` to replace a block in place. Two navigation patterns now exist by design: `HX-Redirect` for "this action takes you somewhere new," in-place swap for "this action updates what you're looking at."
- 2026-07-06: Estimation copy-forward rule (`add_version(copy_estimation=True)`, `app/crm/services.py`): the latest version's Estimation is duplicated as a NEW row attached to the new version; pricing evolves by editing the copy. Estimation rows are never re-parented or shared across versions. Reason: preserves each version's pricing as part of its immutable historical snapshot (D6: estimation prices version with content) while making revisions cheap.
- 2026-07-06: `set_estimation()` validates the MERGED final state (existing values overlaid with passed fields), so switching `pricing_model` requires explicitly passing `None` for now-forbidden fields (e.g. `fixed_price=None` when going FIXED→TIME_BASED). Reason: an update API that silently dropped stale fields could mask caller bugs; the explicit-clear rule keeps every field change visible at the call site. Error messages list missing/forbidden fields by name.
- 2026-07-06: `estimation_is_locked(estimation)` in `app/crm/services.py` is the invariant-#6 seam: it returns False until Phase 4 implements the real bound-to-SENT-offer check. All estimation-mutating services already call it (raising `EstimationLockedError`), so Phase 4 only replaces the hook body — no call sites change.
- 2026-07-06: `ProposalVersion.creator` (`app/crm/models.py`) is a read-only relationship to `core.User` for display ("by {name}" in the version sidebar). Reason: `created_by` was already an FK to `users.id`; a display relationship isn't a cross-domain write, so it doesn't violate CLAUDE.md rule #4 (CRM/Projects write seam) — `core` isn't CRM's sibling domain, it's the auth/base module everything already depends on.
- 2026-07-06: The pricing-model toggle (FIXED/TIME_BASED) on the Estimation form re-renders the field group via an HTMX round trip (`GET .../estimation-fields`), not CSS-only `:checked`/peer selectors. Reason: consistent with every other dynamic interaction in this codebase (all HTMX swaps, zero CSS-only toggles so far), and verifiable by curl — a CSS-only toggle's correctness can only be confirmed in a real browser. Switching models resets the other group's in-progress input (accepted UX tradeoff for a rarely-toggled field).
- 2026-07-06: "Revise from this version" is available on every version's pane, not just the latest — it pre-fills the new version's content from whichever version you're viewing (a restore/branch workflow), while `add_version()`'s copy-forward rule still only ever copies the *latest* version's Estimation, per its Phase 3a contract. The two "source" concepts (content pre-fill vs. estimation copy source) are intentionally decoupled.
- 2026-07-06: At most one open offer (DRAFT or SENT) per lead at a time — `create_offer()` raises `OfferConflictError` otherwise. Reason: prevents parallel contradictory offers on the same lead; a new offer becomes possible only after the open one is DENIED (renegotiation, D4) or ACCEPTED (lead is WON, terminal). Per the Phase 4a kickoff spec.
- 2026-07-06: `deny_offer()` reverts the lead OFFER_SENT → PROPOSAL. Reason: a denial returns the deal to renegotiation (D4); this adds a backward transition to §4.2's forward-only diagram. The revert is conditional on the lead being OFFER_SENT — a lead manually closed (LOST) while its offer was out stays LOST (invariant #10; deny never touches LOST in either direction).
- 2026-07-06: Customers created by `accept_offer()` are always `EXTERNAL`. Reason: pipeline-originated customers are external by definition; INTERNAL customers will be created manually in Phase 5+.
- 2026-07-06: FIXED-price acceptances create the Project with `budget_hours = NULL` (amends §4.4's "internal estimate" wording). Reason: Phase 3a's validation forbids `estimated_units` on FIXED estimations, so there is no unit count to map — the internal cost-control estimate is entered manually on the project later.
- 2026-07-06: DAILY estimation units normalize to hours via `HOURS_PER_DAY = Decimal("8")` (`app/services/offer_acceptance.py`), Decimal arithmetic throughout. Reason: §4.4 requires "estimated_units normalized to hours" but never defined the day length; 8h is the standard assumption, and a module constant keeps it greppable/changeable.
- 2026-07-06: `app/services/offer_acceptance.py` gained two read-only helpers, `project_for_offer()`/`project_for_lead()`, so `app/crm/routes.py` can link a WON lead / accepted offer to its resulting Project without ever importing `app.projects.models` itself. Reason: keeps `app/services/` the *only* module that imports across the CRM/Projects seam (CLAUDE.md #4) even for reads, not just the accept_offer() write path — verified by grep as a Phase 4b acceptance criterion.
- 2026-07-06: `Project.manager` is a read-only relationship to `core.User` (`app/projects/models.py`), same pattern as `ProposalVersion.creator` (Phase 3b) — not a CRM/Projects seam concern, since `core` is the shared auth module every domain already depends on, and ARCHITECTURE.md §3.2 already specifies `manager_id: FK -> User`.
- 2026-07-06: The Offer detail page's status timeline shows "Created" as a bare label with no timestamp, since the `Offer` model (locked in Phase 4a per §3.1) has no `created_at` column — only `sent_at`/`decided_at`. Reason: scope discipline; adding a column now would mean revising an already-committed migration for a phase that's UI-only. If a creation timestamp is ever needed, it's an additive migration, not a Phase 4b concern.
- 2026-07-06: No cascading deletes of project structure: `delete_section`/`delete_work_package` (`app/projects/services.py`) raise `StructureError` while children exist — deletion is bottom-up only, same pattern as the Phase 1 organization delete-block. Reason: structure holds estimates; silent cascade could destroy planning data with one click.
- 2026-07-06: Cross-parent moves (e.g. task → different work package) are out of v1 scope — `move_to_parent()` exists only to raise `StructureError`. v2 candidate. Reason: cross-parent moves need renumbering in two scopes plus UI for target selection; v1's swap-with-neighbor covers the actual reordering need.
- 2026-07-06: Sibling `position` values are contiguous 0..n-1 per parent scope and maintained EXCLUSIVELY by `app/projects/services.py` — there is deliberately no DB unique constraint on (parent_id, position). Reason: a position swap is two UPDATEs that transiently collide; without deferred constraints (which SQLite lacks) a DB unique would reject every move. The kickoff spec explicitly sanctioned service-maintained positions.
- 2026-07-06: `project_rollup()` (`app/projects/queries.py`) returns the WP-level and task-level hour sums independently and they are NEVER coalesced into one number — WP sums are the planning estimate, task sums are the decomposition; divergence is signal, not error. NULLs count as 0 in sums but are returned as separate `unestimated_*` counts so the UI renders them distinguishably. Constant three queries regardless of tree size, enforced by a query-count test.
- 2026-07-06: Budget editing rules (`set_planning_hours`/`update_project_budget`, `app/projects/services.py`): on offer-linked projects, `budget_type`/`over_hours_policy`/`over_rate` are read-only (they record the accepted contract). `budget_hours` on offer-linked projects is editable iff `budget_type` is SOFT — this covers the spec's FIXED case (arrives NULL, set/edited later as internal planning data) and extends the same treatment to TIME_BASED-without-additional-rate acceptances, whose SOFT hours are a reference, not a binding term; on HARD projects the hours come from the accepted estimation and are read-only. Offer-less projects are fully editable with service-level validation mirroring the CHECK constraint (HARD ⇒ hours+policy) plus BILL_AT_RATE ⇒ over_rate; SOFT silently normalizes policy/over_rate to None.
- 2026-07-06: Reordering uses per-item up/down arrow buttons (HTMX swap), not drag-and-drop. Reason: drag-and-drop needs custom JS or a new dependency, both barred by CLAUDE.md rule #6/#5; the arrow pattern maps 1:1 onto `move_item()`'s swap semantics. v2 candidate.
- 2026-07-06: Every structure/budget mutation re-renders one `#project-body` target containing BOTH the budget panel and the structure board. Reason: structure edits change roll-ups and budget edits change the comparison — two independent HTMX targets would go stale against each other (or need out-of-band swaps); one target keeps them consistent by construction. Slightly larger responses, acceptable for a single-user tool.
- 2026-07-06: Manual project creation (`create_project`) sets `manager = current_user` with no picker — single-user system (D3); a manager picker is meaningless until multi-user arrives. `organizations_directory()` in `app/services/offer_acceptance.py` feeds the customer form's org picker so `app/projects/` never imports `app.crm.models`; the import happens lazily inside the route because a module-level import would close a circular chain through the blueprint registration.
